# Требования: Платформа управления трафиком между геораспределёнными дата-центрами
---

## Контекст

Платформа предназначена для управления трафиком между географически разнесёнными дата-центрами (ЦОД) с автоматическим failover и адаптивной балансировкой нагрузки. Система решает проблему ручной настройки балансировщиков, неэффективной маршрутизации между зонами и отсутствия предсказуемого переключения при деградации сервисов. Пользователи — инженеры и платформенные команды, эксплуатирующие распределённые сервисы в мультирегиональной инфраструктуре.

---

## Функциональные требования (FR)

### Обязательные (Must Have)

| ID | Требование | Критерий приёмки |
|---|---|---|
| FR-01 | Система должна предоставлять единую точку входа для всего входящего трафика | Все внешние запросы проходят через Envoy Gateway; прямые обращения к сервисам в обход шлюза невозможны |
| FR-02 | Шлюз должен извлекать идентификаторы из заголовков запросов X-User-ID и X-Cabinet-ID | Запросы с заголовками X-User-ID и X-Cabinet-ID корректно маршрутизируются; значения заголовков доступны downstream-сервисам |
| FR-03 | Платформа должна поддерживать работу 10 сервисов-стабов | 10 стаб-сервисов зарегистрированы и успешно обрабатывают входящие запросы через шлюз |
| FR-04 | Сервисы должны автоматически регистрироваться в системе и обнаруживать друг друга без захардкоженных IP-адресов | При запуске сервис появляется в Consul; сервисы взаимодействуют друг с другом исключительно через service discovery |
| FR-05 | Инфраструктура должна быть развёрнута в двух изолированных средах, симулирующих два географических дата-центра | ЦОД-1 и ЦОД-2 функционируют в изолированных сетях; каждый содержит полный набор сервисов и способен обрабатывать запросы независимо |
| FR-06 | При недоступности сервиса в ЦОД-1 система должна автоматически перенаправлять трафик в ЦОД-2 | Симуляция отказа сервиса в ЦОД-1 → трафик перенаправляется в ЦОД-2 без ручного вмешательства; более 95% запросов обслуживается |
| FR-07 | Должен быть реализован скрипт для генерации трафика к шлюзу | Скрипт запускается одной командой и генерирует нагрузку на Envoy Gateway с настраиваемым RPS |

### Желательные (Should Have)

| ID | Требование | Критерий приёмки |
|---|---|---|
| FR-10 | ML-компонента должна предоставлять рекомендации по оптимальным квотам, лимитам и весам балансировки | По результатам офлайн-анализа метрик компонента генерирует конфигурационные рекомендации, применимые к Envoy |
| FR-11 | Балансировщики в разных ЦОДах должны обмениваться конфигурациями через общий фид в реальном времени | Изменение параметров маршрутизации в одном ЦОД отражается в другом без перезапуска компонентов |

### Не в этом релизе (Out of Scope)

- **Онлайн-инференс ML-компоненты** — динамическая адаптация весов в реальном времени запланирована на v2
- **Поддержка более двух дата-центров** — текущая архитектура рассчитана на топологию из двух ЦОД

## Нефункциональные требования (NFR)

### Производительность

| Метрика | Требование | Как измеряем |
|---|---|---|
| Стабильная нагрузка | ≥ 100 RPS в течение нескольких часов без деградации | Нагрузочное тестирование через скрипт генерации трафика + Grafana |
| Latency (p50) | Отображается на дашборде | Prometheus + Grafana |
| Latency (p95) | Отображается на дашборде | Prometheus + Grafana |
| Latency (p99) | Отображается на дашборде | Prometheus + Grafana |
| Процент успешных запросов при отказе ЦОД | ≥ 95% во время симуляции отказа сервиса | Нагрузочный тест + мониторинг error rate в Grafana |

### Надёжность

| Метрика | Требование |
|---|---|
| Доступность при отказе одного ЦОД | ≥ 95% запросов обслуживается |
| Автоматический failover | Переключение трафика без ручного вмешательства |
| Перезагрузка конфигурации Envoy | Атомарная, без прерывания активных соединений |

### Масштабируемость

- Каждый ЦОД содержит полный и независимый набор сервисов, способный обрабатывать входящие запросы без участия второго ЦОД
- Добавление новых сервисов осуществляется через автоматическую регистрацию в Consul без изменения конфигурации шлюза вручную

### Наблюдаемость

- Должен быть реализован дашборд в Grafana с мониторингом RPS, перцентилей задержки (p50, p95, p99) и доступности каждого сервиса в обоих ЦОДах
- Prometheus собирает метрики производительности со всех компонентов системы
- Jaeger обеспечивает распределённую трассировку запросов через цепочку сервисов

### Совместимость

- Все компоненты контейнизированы и описаны в Docker Compose манифестах
- Топология разворачивается одной командой `docker-compose up` в воспроизводимом окружении

## Ограничения

- **Технологический стек:** backend-сервисы и ML-компонента реализованы на Go + gRPC; межсервисное взаимодействие через Protocol Buffers
- **Инфраструктура:** Docker + Docker Compose (Infrastructure as Code); развёртывание за пределами контейнерного окружения не предусмотрено
- **Топология:** фиксированная двух-ЦОД архитектура (ЦОД-1, ЦОД-2); поддержка N дата-центров — вне скоупа текущей версии
- **ML:** офлайн-анализ метрик; онлайн-инференс в реальном времени — следующая итерация

## Сценарии использования (Use Cases)

### UC-01: Маршрутизация запроса через Envoy Gateway

**Актор:** Внешний клиент (сервис или пользователь)

**Предусловие:** Оба ЦОД развёрнуты и доступны; все сервисы зарегистрированы в Consul

**Основной поток:**
1. Клиент отправляет запрос с заголовками X-User-ID и X-Cabinet-ID на Envoy Gateway ЦОД-1
2. Envoy извлекает идентификаторы из заголовков
3. Envoy проверяет состояние локальных сервисов через Consul health-check
4. Запрос маршрутизируется к целевому сервису внутри ЦОД-1
5. Ответ возвращается клиенту

**Альтернативный поток (failover):**
3а. Целевой сервис в ЦОД-1 недоступен → Envoy автоматически перенаправляет запрос в ЦОД-2  
3б. Запрос обрабатывается аналогичным сервисом в ЦОД-2; ответ возвращается клиенту

**Постусловие:** Запрос обработан; метрики зафиксированы в Prometheus

---

### UC-02: Автоматическая регистрация нового сервиса

**Актор:** Сервис-стаб (автоматически при запуске)

**Предусловие:** Consul доступен в сети ЦОДа

**Основной поток:**
1. Сервис запускается в контейнере
2. Сервис регистрирует себя в Consul с указанием адреса и health-check endpoint
3. Consul начинает периодически проверять состояние сервиса
4. Envoy получает обновлённый список upstream-сервисов из Consul через xDS API
5. Новый сервис начинает принимать трафик через Gateway

**Постусловие:** Сервис доступен для маршрутизации без изменения конфигурации вручную

---

### UC-03: Симуляция отказа ЦОД и проверка failover

**Актор:** Инженер (в рамках нагрузочного тестирования)

**Предусловие:** Оба ЦОД запущены; скрипт генерации трафика активен с нагрузкой 1000 RPS

**Основной поток:**
1. Инженер отключает один или несколько сервисов в ЦОД-1
2. Consul фиксирует деградацию через health-check
3. Envoy автоматически переключает трафик на ЦОД-2
4. Grafana-дашборд показывает кратковременный рост latency и восстановление error rate до нормального уровня
5. Более 95% запросов обслуживается без прерывания

**Постусловие:** Failover подтверждён; метрики сохранены для анализа
