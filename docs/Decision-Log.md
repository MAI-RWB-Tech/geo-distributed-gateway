| DL-T1-001 | 2026-05-21 | Tag-based Consul DNS (`zoneN.<svc>.service.consul`) вместо xDS gRPC discovery | UC-02 в Requirements называет «xDS API», но ADR §5 закрепляет только связку «Envoy + Consul» без обязательства по протоколу discovery. xDS-сервер потребовал бы либо consul-template (отдельный sidecar, лишний failure-mode), либо consul-connect (тянет mTLS, который сам по себе Out-of-Scope). Tag-based DNS — нативный механизм Consul: leading `<tag>.` префикс в `*.service.consul` фильтрует SRV-ответы по тегу. Работает поверх single-DC dev-агента, попадает в TTL-окно ~5s через `respect_dns_ttl` в Envoy-кластере. | DL-T1-002 | 2026-05-21 | Best-effort регистрация в Consul: сервис не падает при недоступном агенте | Стандартный путь — `log.Fatal` при ошибке `client.Agent().ServiceRegister(...)`, тогда контейнер не пройдёт healthcheck и compose отметит depends_on невыполненным. Но в реальности это создаёт хрупкую цепочку: переинициализация Consul (например, при тестах failure-runner) убьёт ВСЕ 10 app-контейнеров. Решение: при пустом `CONSUL_ADDR`, ошибке `NewClient` или ошибке `ServiceRegister` — `slog.Warn` и продолжаем работу. HTTP-сервер слушает `:8080`, healthcheck `/health` отвечает 200, Envoy просто не увидит этот instance в Consul DNS и пропустит его. Trade-off: можно проспать конфигурационный баг (`CONSUL_ADDR` опечатка → silent degrade), но это перевешивается устойчивостью к рестарту control-plane Consul. | T1-impl | ✅ Действует |

| DL-T1-004 | 2026-05-21 | Static IPv4 для Consul (`172.30.{0,1,2}.5`) в каждой Docker-network; `dns_resolvers.address` в zone-envoy = IP, не hostname `consul` | План (plan.md step 5) предписывал `dns_resolvers: [{socket_address: {address: consul, port_value: 8600}}]`. На практике Envoy 1.36 валидирует поле `address` в `dns_resolvers.socket_address` как **IP-литерал**, не hostname: загрузка падает с `malformed IP address: consul`. Резолвер DNS не может сам себя зарезолвить через DNS — это chicken-and-egg. Решения два: (а) static IP через ipam/ipv4_address (выбрано), (б) hardcode IP получаемый из `docker inspect` (нестабильно при пересоздании сети). Реализация: в `networks.{global,zone1,zone2}-net` добавлены `ipam.config.subnet` (`172.30.{0,1,2}.0/24`); Consul прибит к `.5` в каждой подсети; zone1-envoy.yaml использует `172.30.1.5`, zone2-envoy.yaml — `172.30.2.5`. Trade-off: фиксированные подсети могут конфликтовать с другими compose-проектами на той же машине; компромисс приемлем для dev-стенда. | T1-impl + @mixxxaile | ✅ Действует |
| DL-T1-005 | 2026-05-21 | Регистрация в Consul по фактическому IPv4 контейнера, а не по container_name | План (plan.md step 4) указывал `Address: instance` (container_name) и комментарий «Docker DNS резолвит container_name → IP». На практике Consul DNS возвращает запись как **CNAME** (canonical name → container_name), а Envoy с `dns_resolvers: [Consul]` после получения CNAME не имеет второго резолвера для следования за ним → `service-X-zoneN-cluster` остаётся без endpoints, global-envoy отдаёт 503. Решение: helper `firstNonLoopbackIPv4()` через `net.InterfaceAddrs()` находит IP контейнера на старте, и регистрация передаёт его в `Address`. Consul DNS тогда возвращает A-record напрямую. Fallback на `instance` сохранён на случай аномальной сетевой конфигурации. | T1-impl + @mixxxaile | ✅ Действует |
| DL-T4-001 | 2026-05-22 | Prometheus self-метрики экспонируются вручную (text exposition format) без `prometheus/client_golang` | Контракт `/metrics` требует только 3 метрики (gauge + counter с одним label + gauge). Подключение `prometheus/client_golang` тянет 5+ транзитивных зависимостей (`prometheus/common`, `prometheus/procfs`, `golang/protobuf`, `cespare/xxhash`, `prometheus/client_model`), что для бинарника с пустым `go.sum` сегодня — несоразмерно. Hand-rolled text format в `handleMetrics` использует только stdlib (`fmt`, `strconv`). Trade-off: если позже понадобятся histogram/summary — придётся либо переписать (несложно), либо завести зависимость. Сегодня выигрыш в скорости сборки и минимальном attack surface перевешивает. | T4-impl | ✅ Действует |
| DL-T4-002 | 2026-05-22 | `/healthz` 200 = «последний успешный pull был недавно (≤ 2.5 × poll-interval)», а не «когда-либо был успешный pull» | План T4 step 5 формулировал: «200 если был хотя бы один успешный pull (version > 0)». Acceptance scenario A в том же плане требует: «при остановке ML и ожидании > 2 × poll-interval → /healthz = 503». Это явное противоречие: с буквальным `version > 0` healthz никогда не флипнется обратно в 503. Реализован компромисс через CLI-флаг `-health-max-age-factor` (default `2.5`): 200, если `version > 0` И `now - lastSuccess ≤ factor × poll_interval`; 503 иначе. Factor 2.5 (а не 2.0) даёт небольшой запас на jitter и сетевой ретрай в одном poll-цикле. Это даёт реальный liveness-сигнал (отличает «ML свежие данные» от «ML давно отвалилось»). `/config` продолжает отдавать last known good снапшот даже когда `/healthz=503` — graceful degradation сохраняется. | T4-impl | ✅ Действует |
| DL-T4-003 | 2026-05-22 | `replace` директива в `cmd/control-plane/go.mod` без соответствующего `require` | План step 1: «replace добавляется сразу — несмотря на то, что T4 sdk не использует, T7 будет публиковать `sdk/config.RoutingHints` через этот модуль». Стандартный путь (`require sdk v0.0.0` + `replace`) ломается на `go mod tidy`: tidy удаляет require без импорта, и хоть replace останется — visually связь теряется. Альтернатива через blank-import (`_ "github.com/geo-distributed-gateway/sdk/config"`) — добавляет ненужный init() и формальную транзитивную зависимость. Выбран минималистичный вариант: `replace` директива без `require`. Это валидный Go syntax (replace без require — inert до первого импорта), tidy её не удаляет. Когда T7 добавит import, tidy сам добавит require и replace начнёт действовать. Комментарий в go.mod объясняет намерение. | T4-impl | ✅ Действует |
| DL-T4-004 | 2026-05-22 | Первый успешный pull всегда устанавливает version = 1, даже если ML вернул payload deep-equal начальному пустому состоянию | План: «если содержимое отличается от предыдущего успешного снапшота — инкрементируй version на 1». Начальный пустой `Snapshot{Version:0, Weights:{}, RateLimits:{}}` не является «успешным снапшотом» — это placeholder для graceful degradation. Если бы ML вернул действительно пустой payload (`{weights:{}, rate_limits:{}}`), `reflect.DeepEqual` дал бы true, и version остался бы 0 → `/healthz` навсегда 503 при стабильном пустом ML. Спец-кейс: если `prev.Version == 0` после успешного pull, форсируем `next.Version = 1`. Это совпадает с acceptance scenario A («version ≥ 1 после первого pull») и не влияет на нормальный кейс (ML возвращает 5 сервисов → DeepEqual=false → version=0+1=1). | T4-impl | ✅ Действует |
| DL-T6-001 | 2026-05-22 | Sentinel `x-route-fallback: true` НЕ эмитится. Trampoline через `dynamicMetadata` (план T6 step 1) не работает в Envoy 1.36 Lua bindings. Fallback виден через warn-логи global-envoy. | План step 1 предложил trampoline: на request-side `streamInfo():dynamicMetadata():set(ns, k, v)`, на response-side `get(ns)` → `headers:add("x-route-fallback","true")`. Эмпирически в Envoy 1.36 (verified `docker logs global-envoy`): `dynamicMetadata():get(ns)` возвращает `nil` на response-phase даже после успешного `set` на request-phase (нет ошибки, тихая потеря данных). Попытка обойти через round-trip request-header после `httpCall(...)` падает с `object used outside of proper scope` — Envoy инвалидирует headers-handle после httpCall. Filter-state API в Lua bindings либо отсутствует, либо требует `StateType`/`LifeSpan` enum-ов, не экспортированных в Lua. Workaround: `envoy_on_response` no-op, fallback виден только через warn-логи (уже эмитятся в `fetch_weights`: `score: control-plane unavailable for <svc> (status=...) → fallback`). Acceptance-проверка sentinel переформулирована: вместо `curl -sI ... grep x-route-fallback` использовать `docker logs global-envoy 2>&1 \| grep "score:.*fallback"`. Q-T6-001 закрыт (выбран этот workaround); plan.md правка acceptance отложена до следующего ревью документации. | T6-impl + orchestrator + @mixxxaile | ✅ Действует |
| DL-T6-002 | 2026-05-22 | Per-worker Lua cache, кешируется ТОЛЬКО успешный ответ Control Plane (не fallback) | Тривиальная реализация — кешировать любой результат `fetch_weights` на TTL=30s. Проблема: при 5-секундной недоступности CP один промах кеша → закешировали `{0.5, 0.5}` на 30s, и даже после восстановления CP трафик ещё 25s балансируется 50/50 вместо реальных весов. Решение: ветка `if not was_fallback then M.cache[service] = ... end` — fallback не запоминается, следующий запрос снова дёрнет CP. При долгой недоступности CP это даёт N×httpCall на каждый промах (нет single-flight, Lua не имеет async-примитивов), но т.к. CP в любом случае не отвечает — лишний httpCall стоит ~таймаут (1s) и завершается уходом в fallback. Trade-off: при флапающем CP (success/fail на каждом 2-м запросе) кеш будет постоянно перестраиваться вместо буфера. Приемлемо для dev-стенда. | T6-impl | ✅ Действует |
| DL-T6-003 | 2026-05-22 | RNG seed = `os.time() + tonumber(tostring({}):sub(8))` — per-worker энтропия из адреса пустой таблицы | Без `math.randomseed` стандартная Lua-реализация даёт детерминированную последовательность `math.random()` на каждый свежий Lua state. У Envoy ~2-4 worker-thread, каждый со своим Lua state — все они дали бы идентичные «случайные» выборы зон для идентичных входных условий, что для weighted random — сильное искажение распределения. `os.time()` (секундная гранулярность) сам по себе мог бы совпадать между worker-ами (старт почти одновременно). Хак `tostring({}):sub(8)` извлекает hex-часть адреса свежесозданной таблицы — гарантированно уникальна per-worker per-load (Lua-память отдельная). Альтернатива через `/dev/urandom` потребовала бы LuaJIT FFI; через C-extension — добавила бы зависимость. Идиомы в Envoy-сообществе — этот же hack. | T6-impl | ✅ Действует |
